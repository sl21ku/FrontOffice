# 実装計画書: UI/わかりやすさ改善20項目

## 目的

この計画書は、シミュレータの制度や数値を増やしても、ユーザーが迷わず遊べるようにするためのUI/UX改善計画である。

現在の方向性は「NPB風のガチ球団運営」だが、制度を増やすほど画面は複雑になる。  
そのため、以下の3点を重視する。

- 今やるべき操作がすぐ分かる
- 選手・球団・制度の意味がすぐ分かる
- 補強や放出の影響が事前に分かる

## 優先度

| 優先度 | 意味 |
| --- | --- |
| P0 | 迷いやミスを減らすため最優先 |
| P1 | ゲーム性と納得感を大きく上げる |
| P2 | 長期プレイ、分析、演出を強化する |
| P3 | 仕上げ、快適性、補助説明 |

## 推奨フェーズ

| Phase | 対象 | 狙い |
| --- | --- | --- |
| Phase 1 | 進行バー、TODO、警告色、ツールチップ | 迷わない土台を作る |
| Phase 2 | 一軍登録、弱点診断、補強プレビュー、選手詳細 | 編成判断を分かりやすくする |
| Phase 3 | 比較、ドラフトボード、契約更改、FA競合 | ガチ編成ゲームとして深める |
| Phase 4 | GM評価、球団方針、ファン/オーナー、ニュース | 長期プレイの物語を作る |
| Phase 5 | ポストシーズン、二軍/育成、トレード提案 | シーズン全体の体験を厚くする |

## 1. オフシーズン進行バー

優先度: P0

目的:
ユーザーが「今どの工程にいるか」「次に何をすればいいか」を即座に把握できるようにする。

画面案:
- 画面上部、トップバー下に横長のステップバーを置く。
- ステップ例:
  `戦力確認 -> 戦力外 -> FA -> 現役D -> ドラフト -> 一軍登録 -> 開幕`
- 完了済みは緑、現在位置は濃色、未完了は薄色で表示する。
- クリックで該当タブへ移動できるようにする。

実装方針:
- `state.offseasonStep` を追加するか、既存の `activeTab` と状態から自動判定する。
- `renderOffseasonProgress()` を作り、`renderGame()` 内の上部に配置する。
- FA交渉中、ドラフト中、現役ドラフト中は該当ステップを強調する。

確認観点:
- 開始直後に「戦力確認」が現在位置になること。
- ドラフト中は「ドラフト」が現在位置になること。
- 開幕後は進行バーが結果確認モードになること。

## 2. やることリスト

優先度: P0

目的:
開幕前に必要な作業や警告を自動で出し、ユーザーの見落としを防ぐ。

画面案:
- サイドバーまたはメイン上部に「やること」パネルを出す。
- 例:
  - 支配下70人超過
  - 一軍登録31人未満
  - 外国人枠超過
  - ドラフト未実施
  - FA交渉中
  - 契約更改未完了
- 各項目に「移動」ボタンを付ける。

実装方針:
- `getActionItems()` を作る。
- 各制度状態を読み取り、配列で `{ severity, label, tab, message }` を返す。
- `renderActionItems()` で表示する。
- 開幕ボタンの有効/無効判定にも同じ関数を使う。

確認観点:
- 問題がないときは「開幕できます」と表示されること。
- 問題があるときに開幕ボタン付近でも警告が見えること。

## 3. 一軍登録画面

優先度: P0

目的:
支配下67〜70人から一軍31人を選ぶ判断を、ユーザー操作として成立させる。

画面案:
- 左に支配下選手一覧、右に一軍登録リスト。
- 上部にカウンター:
  - 一軍 `31/31`
  - ベンチ `26/26`
  - 投手数
  - 捕手数
  - 外国人枠
- 選手カードに「登録」「抹消」ボタン。
- 自動編成ボタンも用意する。

実装方針:
- `state.activeRosterIds[teamId]` を追加する。
- 未設定時は既存の `selectActiveRoster()` を使う。
- `renderActiveRosterPanel()` を新規作成する。
- 31人超過、外国人枠超過、捕手不足などは警告する。

確認観点:
- 31人を超えて登録できないこと。
- 登録外選手に一軍成績が付かないこと。
- 保存/ロード後も一軍登録が維持されること。

## 4. チーム弱点診断

優先度: P1

目的:
ユーザーが補強ポイントを自分で探しやすくする。

画面案:
- サイドバーまたは戦力タブに「弱点診断」を表示する。
- 例:
  - 先発不足
  - 捕手層が薄い
  - 二遊間の守備が弱い
  - 外国人野手過多
  - 若手不足
  - 年俸効率が悪い

実装方針:
- `analyzeTeamNeeds(teamId)` を追加する。
- ポジション別人数、平均能力、年齢構成、年俸、外国人枠を評価する。
- 診断結果に `severity` と `recommendedAction` を持たせる。

確認観点:
- 投手が少ないチームで先発/中継ぎ不足が出ること。
- 捕手1人編成で捕手不足が出ること。

## 5. 補強効果プレビュー

優先度: P1

目的:
FA、トレード、ドラフト候補を獲得した場合の戦力変化を事前に見せる。

画面案:
- 選手カードの下部に「獲得後プレビュー」を表示する。
- 例:
  - 総合力 `+2`
  - 打撃 `+4`
  - 先発 `変化なし`
  - 支配下 `68/70`
  - 予算残 `-1.2億`
- トレードでは放出後/獲得後を比較表示する。

実装方針:
- `previewRosterChange(teamId, addPlayers, removePlayers)` を作る。
- 仮想ロスターを作って `teamStrength()` 相当を計算する。
- FA/トレード/ドラフトカードから呼び出す。

確認観点:
- 獲得ボタンを押す前に戦力差分が見えること。
- 支配下上限や予算不足がプレビューで分かること。

## 6. 年俸/予算の見える化

優先度: P1

目的:
球団運営のコスト構造を分かりやすくし、年俸管理をゲーム性にする。

画面案:
- 予算パネルに横棒グラフを表示する。
- 分類:
  - 主力
  - 控え
  - 若手
  - 外国人
  - FA獲得選手
- 高額選手ランキングも表示する。

実装方針:
- `getPayrollBreakdown(teamId)` を追加する。
- 年齢、総合、外国人、獲得経路で分類する。
- CSSのみで簡易バーを作る。

確認観点:
- 年俸総額と内訳合計が一致すること。
- 高額外国人やFA選手が分類されること。

## 7. 選手カードのタブ化

優先度: P1

目的:
選手詳細が縦に長くなりすぎるのを防ぎ、必要情報を探しやすくする。

画面案:
- 選手詳細モーダルにタブを追加する。
- タブ:
  - 概要
  - 今季成績
  - 通算成績
  - 能力
  - 契約/FA
  - 起用/登録

実装方針:
- `state.playerDetailTab` を追加する。
- `renderPlayerDetail()` を小さな関数へ分割する。
- `renderPlayerOverviewTab()`, `renderPlayerStatsTab()` などを作る。

確認観点:
- タブ切替でモーダルが閉じないこと。
- モバイルでもタブが横スクロールで操作できること。

## 8. 選手比較機能

優先度: P1

目的:
FA獲得、トレード、戦力外、一軍登録の判断をしやすくする。

画面案:
- 選手一覧に「比較」チェックを追加。
- 2人選ぶと比較モーダルを表示する。
- 能力、年俸、年齢、成績、契約、FA年数を横並びにする。
- 差分は緑/赤で表示する。

実装方針:
- `state.comparePlayerIds` を追加する。
- `renderCompareModal()` を作る。
- 2人以上選んだ場合は最初の2人だけ比較、または最大3人対応にする。

確認観点:
- 同一ポジション比較が分かりやすいこと。
- 投手と野手を比較しても表示が破綻しないこと。

## 9. ドラフトボード

優先度: P1

目的:
ドラフト候補が増えても、狙いの選手を管理できるようにする。

画面案:
- 候補カードにタグ付け:
  - お気に入り
  - 1位候補
  - 下位候補
  - 地雷注意
  - 即戦力
  - 素材型
- ボード表示でタグ別に候補を一覧する。

実装方針:
- `state.draftBoardTags[playerId] = []` を追加する。
- タグボタンを候補カードに付ける。
- `renderDraftBoardView()` を追加する。

確認観点:
- タグが保存/ロードされること。
- 1巡目抽選後もタグが維持されること。

## 10. ニュースログの強化

優先度: P2

目的:
長期プレイで「何が起きたか」を振り返れるようにする。

画面案:
- ニュースをカテゴリ別に色分けする。
- カテゴリ:
  - 移籍
  - ドラフト
  - 成績
  - 引退
  - 契約
  - ポストシーズン
- 年度フィルターを付ける。

実装方針:
- ニュースを文字列配列から `{ year, category, text, importance }` に移行する。
- 既存ニュースは読み込み時に互換変換する。
- `renderNewsPanel()` を強化する。

確認観点:
- 重要ニュースが消えずに履歴として残ること。
- 古いセーブも表示できること。

## 11. GM評価レポート

優先度: P1

目的:
シーズン後にユーザーの編成判断を評価し、次のプレイ目標を作る。

画面案:
- 結果画面に「GM評価」カードを表示する。
- 指標:
  - 補強成功度
  - 年俸効率
  - 若手起用
  - 守備改善
  - ファン満足度
  - オーナー評価
- S/A/B/C/Dで総合評価。

実装方針:
- `evaluateGMPerformance(teamId, result)` を追加する。
- 前年比、順位、予算、若手出場、FA成功などからスコアを作る。
- `state.history` に評価を保存する。

確認観点:
- 優勝しても年俸超過なら満点にならないなど、評価に幅が出ること。

## 12. 球団方針の選択

優先度: P1

目的:
同じ順位でも、球団方針によって成功/失敗の意味を変える。

画面案:
- オフ開始時に方針を選択する。
- 方針:
  - 優勝狙い
  - 再建
  - 若手育成
  - 財政健全化
  - 守備重視
- 選択した方針はステータスに表示する。

実装方針:
- `state.teamPolicy` を追加する。
- GM評価、ファン満足度、補強おすすめに反映する。
- 方針ごとに評価重みを変える。

確認観点:
- 再建方針なら若手起用が高評価になること。
- 優勝狙いなら順位が強く評価されること。

## 13. ファン・オーナーの反応

優先度: P2

目的:
補強や放出に感情的な反応を出し、球団運営の手触りを増やす。

画面案:
- サイドバーに「ファン」「オーナー」評価を表示する。
- 主力放出でファン低下、財政改善でオーナー上昇など。
- ニュースに短い反応コメントを出す。

実装方針:
- `state.fanTrust`, `state.ownerTrust` を追加する。
- 戦力外、FA獲得、トレード、順位結果で変動させる。
- 結果画面に前年差を表示する。

確認観点:
- 人気選手放出時にファン評価が下がること。
- 高額補強失敗時にオーナー評価が下がること。

## 14. 契約更改画面

優先度: P1

目的:
年俸調整を単なる手入力ではなく、シーズン後の重要フェーズにする。

画面案:
- シーズン終了後に「契約更改」タブを表示する。
- 選手ごとに現在年俸、希望年俸、提示額、反応を表示する。
- 一括更改、自動更改、個別交渉を用意する。

実装方針:
- `state.contractRenewals` を追加する。
- `generateRenewalDemands()` で希望額を作る。
- `submitRenewalOffer(playerId, amount)` を実装する。

確認観点:
- 全員の契約更改が終わらないと翌年へ進めないこと。
- 主力の大幅昇給が予算に響くこと。

## 15. トレード提案UI

優先度: P2

目的:
ユーザーが探すだけでなく、他球団から提案が来ることでゲームが動くようにする。

画面案:
- トレードタブに「他球団からの提案」欄を追加する。
- 提案カードに放出/獲得、成立理由、拒否/交渉/承諾ボタンを出す。

実装方針:
- `generateTradeOffers(teamId)` を追加する。
- CPU球団の弱点と余剰ポジションから提案を作る。
- `state.tradeOffers` に保存する。

確認観点:
- 明らかに不公平な提案ばかりにならないこと。
- 提案承諾時に70人枠チェックが効くこと。

## 16. FA交渉の競合演出

優先度: P1

目的:
FA交渉に駆け引きと緊張感を出す。

画面案:
- FA交渉モーダルに「他球団提示予測」を表示する。
- 例:
  - 他球団A: 3年 2.4億
  - 他球団B: 2年 2.8億
- 成功率をバーで見せる。

実装方針:
- `generateCompetingFAOffers(player)` を追加する。
- 交渉モーダル表示時に競合提示を生成する。
- 提示額・年数・球団格・地元要素で成功率を計算する。

確認観点:
- 高額提示でも年数が短いと負けるケースがあること。
- 自球団FAには残留補正が見えること。

## 17. ポストシーズン演出

優先度: P2

目的:
CS/日本シリーズを単なるニュースではなく、シーズンの山場として見せる。

画面案:
- 結果画面にトーナメント表を表示する。
- CSファースト、ファイナル、日本シリーズを段階表示。
- 優勝チームは大きく表示する。

実装方針:
- ポストシーズン実装後、`renderPostseasonBracket()` を追加する。
- `state.results.postseason` を読み取って描画する。

確認観点:
- 操作球団が敗退した場合も、その後の勝者が追えること。

## 18. 二軍・育成画面

優先度: P1

目的:
若手育成と支配下昇格を分かりやすくする。

画面案:
- 「二軍/育成」タブを追加する。
- 表示内容:
  - 二軍成績
  - 育成選手一覧
  - 支配下昇格候補
  - 怪我明け選手
  - 若手期待度
- 支配下登録ボタンを置く。

実装方針:
- 育成制度・二軍成績実装後に追加する。
- `renderFarmPanel()` を作る。
- 昇格候補は成績、年齢、潜在能力でソートする。

確認観点:
- 育成選手が支配下人数に含まれないことが画面上でも分かること。

## 19. 警告カラーの整理

優先度: P0

目的:
UI全体の意味づけを統一し、色を見るだけで状態が分かるようにする。

画面案:
- 色ルール:
  - 赤: 上限超過、不可、重大警告
  - 黄: 注意、未完了、リスクあり
  - 緑: 適正、完了、成功
  - 青: 情報、成長株、候補
  - 金: 注目、上位候補、タイトル

実装方針:
- CSSに `.tag.warning`, `.tag.success`, `.tag.info`, `.tag.danger` を追加する。
- 既存の赤/金タグを整理する。
- `severity` を持つUI表示は共通関数で色を決める。

確認観点:
- 同じ意味の警告が画面によって違う色にならないこと。

## 20. 初心者向けツールチップ

優先度: P0

目的:
制度名が多くても、ユーザーが外部知識なしで理解できるようにする。

画面案:
- `人的補償`, `FAランク`, `支配下`, `現役ドラフト`, `ポスティング`, `一軍登録`, `育成` などに `?` アイコンを付ける。
- ホバーまたはタップで短い説明を表示する。

実装方針:
- `HELP_TEXTS` 定数を作る。
- `renderHelpTip(key)` を作る。
- PCはホバー、スマホはタップで開閉できるようにする。

確認観点:
- スマホでも説明が読めること。
- 説明文が長すぎず、画面を塞がないこと。

## 完了条件

この計画の実装が完了した状態では、以下を満たすこと。

- 初見ユーザーが次に何をすべきか迷わない
- 補強、放出、登録の結果が事前に分かる
- 制度名の意味が画面内で説明される
- 一軍/二軍/育成/支配下の違いがUIで明確に分かる
- 長期プレイで過去のニュース、評価、方針が追える
- モバイル幅でも主要操作が破綻しない
